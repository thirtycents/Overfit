AI 评测体系：面试口语版
每个概念用一段话说清楚，可以直接在面试时讲出来。附 2025-2026 工业界最新动态。

---
一、Agent 评测框架
AgentBench
AgentBench 是清华大学做的，专门评测大模型作为 Agent 能不能真的完成任务。它设计了 8 种真实环境——比如让模型去操作命令行、查数据库、刷网页购物——然后看模型能不能多轮交互下来把任务完成。核心指标就是任务成功率，说白了就是"完成了多少任务"。它最大的发现是，闭源模型（比如 GPT-4）和开源模型在 Agent 任务上的差距，远比在单轮问答上的差距大得多。
τ-bench（tau-bench）
τ-bench 是 Sierra AI 做的，专门模拟真实业务 Agent，比如航司客服、电商退款这种场景。它最大的特点是用另一个大模型来扮演"动态用户"，而不是写死的脚本，所以每次对话都不一样，测的是 Agent 在不确定性下的稳定性。它的核心指标叫 pass^k，就是把同一个任务跑 k 次，看每次都能成功的概率——这个和 pass@k 不一样，pass@k 是"k 次里至少成一次"，τ-bench 的 pass^k 要求是"每次都得稳"，更严格。
SWE-bench（工业界最爱，2025 年度最重要评测）
SWE-bench 是现在工业界最广泛引用的代码 Agent 评测，拿 GitHub 上真实的 Bug Issue 让模型去修，修完之后跑仓库的自动化测试，看 PR 能不能通过。它的核心指标是 Resolved Rate（解决率）。2023 年最好的模型才能解决 4.4%，到 2024 年底就飙到了 71%，2025 年 Claude Opus 4.5 在 SWE-bench Verified 上达到了 80.9%，现在各家公司发布新模型都必须报这个数。
OSWorld
OSWorld 是让模型直接操作真实电脑桌面（Linux/macOS/Windows 的 GUI），截图给模型看，让它点击、输入、拖拽，完成 369 个真实任务，比如"把这个文件转成 PDF 发送"。它测的是多模态 + 规划 + 执行的综合能力，目前最难，2025 年最好的模型也才能完成 30% 左右，是业界公认的下一个难关。
BFCL（Berkeley Function-Calling Leaderboard）
BFCL 是 UC Berkeley 专门测工具调用（Function Calling）能力的榜单，给模型看 API 文档和用户需求，看它能不能正确选函数、填对参数。这个在工业界非常实用，因为现在绝大多数 Agent 产品都依赖 Tool Call，BFCL 直接反映"这个模型能不能在我的 API 上稳定工作"。顶级模型现在能达到 85-90% 的准确率，差的开源模型会掉到 60% 以下。
GAIA
GAIA 是 Meta AI 和 HuggingFace 一起做的，让模型完成真实世界里的助理任务，比如"帮我找 2023 年某个会议的注册费是多少"，需要模型自己搜索、看图、推理、调工具。分三个难度等级，难度越高需要的工具调用链越长。特点是答案有标准答案可以精确比较，不用人工打分。

二、Agent 核心指标
Tool Call Accuracy（工具调用准确率）
这个指标衡量的是模型在对的时机调对的工具、填对参数的能力。通常拆成三层来看：第一层是工具选择准确率，就是模型有没有选对工具；第二层是参数准确率，就是参数名对不对、类型对不对、值对不对；第三层是调用序列准确率，多个工具有没有按正确顺序调用。工业界在做内部评测的时候一般这三层都要分开报，因为模型可能选对工具但参数填错，或者工具选错但参数格式正确，定位问题需要分开看。
pass@k（代码生成）
pass@k 是 OpenAI 在 2021 年 Codex 论文里提出的，意思是：对同一道题采样 k 个答案，只要有一个能通过测试就算成功，这道题的概率是多少。k=1 就是单次通过率，k=10 说明给模型 10 次机会，k=100 是能力上界。实际计算时为了统计无偏，会采样 n 个（比如 200 个），用组合数公式来估计。面试时说清楚"pass@k 是至少一次成功，τ-bench 的 pass^k 是每次都要成功"就能体现理解深度。
Task Success Rate vs. Partial Success
任务成功率（TSR）是最直观的指标，就是完全成功的任务比例。但现在工业界越来越多用**部分成功（Partial Success）**来补充，因为一个 20 步的任务完成了 18 步和完成了 0 步，如果只看 TSR 都是失败，但差距很大。具体实现上常见的有两种：一种是基于规则的状态验证（检查数据库/文件系统是否达到目标状态），另一种是 LLM-as-Judge（让强模型来打分）。

三、大模型通用评测
MMLU（知识广度，但已饱和）
MMLU 是涵盖 57 个学科的选择题测试，从高中到研究生水平都有，一共 14000 多道题，4 选 1。它是过去几年最常引用的通用知识评测，但现在顶级模型都已经超过 88% 了，已经饱和，区分度很低。2025 年工业界基本不拿它来区分顶级模型了，更多用来看中低端模型的基础能力。
GPQA Diamond（2025-2026 工业界新宠）
GPQA 是由 PhD 专家出的生物、化学、物理选择题，专门设计成"Google 也搜不到答案"的难度，领域外的 PhD 拿着互联网也只有 34% 的正确率。GPQA Diamond 子集是目前区分顶级模型最有效的推理基准，2026 年 2 月 Gemini 3.1 Pro 达到了 94.3%，Claude Opus 4.6 是 91.3%。它之所以重要是因为既测推理深度，又测知识边界，和实际生产环境中模型的表现相关性比 MMLU 高得多。
LiveCodeBench（代码评测，防污染）
LiveCodeBench 持续从 LeetCode、AtCoder、Codeforces 上抓新题，因为这些题是模型训练截止日之后才出现的，所以能有效防止训练数据污染。相比 HumanEval（164 道题，顶级模型已经 93%+，已饱和），LiveCodeBench 更能反映模型真实的代码能力。工业界现在做代码模型对比时基本都会同时报 SWE-bench 和 LiveCodeBench 两个。
Chatbot Arena / LM Arena（人类偏好的黄金标准）
Chatbot Arena 是 UC Berkeley 做的，让真实用户盲测两个模型的回答然后投票，用 Bradley-Terry 模型把投票转成 ELO 分。它的优势是覆盖了自动评测发现不了的主观质量，比如表达是否自然、逻辑是否流畅。缺点是成本高、慢、可能有冗长偏见（长答案容易赢）。2025 年底 Gemini 3 Pro 在 Arena 总榜排第一，ELO 1501 分。工业界现在默认"Chatbot Arena 高 = 用户真的觉得好用"。
MT-Bench（多轮对话质量）
MT-Bench 是 LMSys 做的，80 道多轮对话题，覆盖写作、推理、数学、代码等 8 类，每道题会追问一轮，然后用 GPT-4 作为评委打 1-10 分。这是"LLM-as-Judge"模式的代表性做法。面试时可以说：它的意义在于第一次规模化地用强模型当评委，解决了开放式生成任务没法自动打分的问题，但也引入了"评委模型本身的偏见"这个新问题。
AIME / MATH-500（数学推理）
AIME 是美国高中数学邀请赛真题，MATH-500 是竞赛数学精选题。GSM8K（小学数学）已经被顶级模型刷满了，AIME 2025/2026 是现在区分推理模型最有效的数学基准。2026 年初，Qwen3.5-plus 在 AIME 2026 上达到 91.3%，说明现在顶级开源模型的数学能力已经接近闭源模型了。

四、强化学习评测
Atari + Human Normalized Score
Atari 57 款游戏是 RL 最经典的评测环境。核心指标是 Human Normalized Score（HNS），计算方式是：（Agent 得分 - 随机策略得分）÷（人类平均得分 - 随机策略得分）× 100%。HNS = 100% 表示和人类一样好，超过 100% 就是超越人类。报告时通常取 57 个游戏的中位数 HNS，因为均值容易被少数爆分游戏拉高。
MuJoCo 连续控制
MuJoCo 是做机器人控制的标准仿真环境，有 HalfCheetah、Hopper、Walker2d 等任务，全部是连续动作空间。指标就是 Episode Return（一个回合的累计奖励），通常取最后 100 个 episode 的均值报告。这个场景的核心挑战是样本效率：同样的性能目标，TD3/SAC 这类 off-policy 算法需要的数据远少于 on-policy 的 PPO。
多种子实验规范（面试必说）
RL 评测里最容易被问到的工程细节是多种子规范。RL 对随机种子极其敏感，同一个算法跑不同种子，最终性能可能差一倍。所以规范做法是至少跑 5 个随机种子（推荐 10 个），必须报均值 ± 标准差，不能只报最好那次。更严格的论文还会用 IQM（四分位均值）来代替均值，因为 IQM 对极端值更鲁棒。
D4RL 离线 RL 评测
D4RL 是离线 RL 的标准数据集，数据质量分四档：random（随机策略）、medium（中等策略）、expert（专家策略）、mixed（混合）。核心指标是归一化得分：(Agent 回报 - 随机回报) ÷ (专家回报 - 随机回报) × 100，满分 100 表示和专家一样好。离线 RL 最大的挑战是"分布外泛化"，模型只见过历史数据里的行为，要推广到没见过的状态时容易崩溃，评测时要特别关注 medium 和 mixed 数据集上的性能，因为这两档最接近真实的工业数据质量。
RLHF 奖励模型评测（RewardBench）
训 RLHF 的核心是奖励模型（RM），RM 的质量直接决定 PPO/DPO 训练的上限。RewardBench 是 AllenAI 做的 RM 评测基准，核心指标是配对准确率：给 RM 一个好回答和一个坏回答，看它能不能正确把好的排得更高。配对准确率越高说明 RM 的人类偏好对齐越好。评测分四类：普通对话、困难对话、安全场景、推理场景，因为 RM 在不同场景下表现差异很大。

五、微调评测
三维评测矩阵（面试必说框架）
做微调评测，不能只看目标任务上的提升，要用三维矩阵：第一维是目标任务性能，看微调是否有效；第二维是通用能力保持，看 MMLU、GSM8K 这些基础能力有没有退化；第三维是灾难性遗忘检测，看安全性有没有变差。很多工程团队只看第一维，结果微调后模型虽然在垂直任务上强了，但通用推理能力和安全性都下降了，这是常见的踩坑点。
灾难性遗忘（Catastrophic Forgetting）
灾难性遗忘是指模型在学新任务的时候把旧知识"覆盖"掉。评测方式是：微调前后在通用基准上跑同一套评测，对比差值。常见警戒线：MMLU 下降不超过 2%，代码能力不超过 3%，MT-Bench 不超过 0.3 分。出现遗忘的常见原因是学习率太高或者微调数据集分布太窄。解决方案有两种：一是数据回放（在微调数据里混入通用指令数据）；二是参数高效微调（LoRA 等方法只动一小部分参数，破坏性更小）。
LoRA 评测关注点
LoRA 的核心超参是秩 r，r 越大，可训练参数越多，学习能力越强，但也越容易过拟合。评测时一般做消融实验，跑 r=4/8/16/32/64 几档，看性能曲线在哪里收敛。实践中 r=16 是一个常用甜点，参数量只占总参数的 0.1%-0.5% 左右，效果往往和 Full Fine-tuning 差不多。另一个要关注的是 alpha/r 的比值，通常设为 1 或 2，比值太大模型容易不稳定。
pass@k 在微调评测中的用法
微调代码模型时，pass@k 是标准指标。特别要注意的是，pass@1 反映的是"有多稳"，pass@10 或 pass@100 反映的是"能力天花板在哪"。有时候微调后 pass@1 下降了但 pass@10 上升了，说明模型学会了一种更有效但方差更大的解法，需要结合 temperature 设置一起分析。

六、工业界 2025-2026 最新趋势
基准饱和问题（最热门话题）
MMLU、GSM8K、HumanEval 这些老基准都已经基本饱和了，顶级模型在上面的差异只有 1-2%，根本分不出高下。2023 年引入的 GPQA、MMMU、SWE-bench 这些更难的基准，到 2024 年性能提升非常显著，但部分也在快速逼近天花板。目前工业界的应对策略有两个：一是持续动态更新（比如 LiveBench 每月出新题）；二是转向真实任务评测（直接在生产数据上跑人类偏好评估）。
LLM-as-Judge 成为主流
LLM-as-Judge 使用大模型比如 GPT-4 来评估回答，大致模仿人类的判断方式，现在已经是工业界评测开放式生成任务的标配。2025 年工业界的共识是：LLM-as-Judge 要做位置随机（交换 A/B 顺序取均值） 来消除位置偏见，要做长度控制来消除冗长偏见，要在人类偏好数据上做校准来验证评委模型本身的可靠性。
SWE-bench Verified 成为代码 Agent 必报指标
SWE-bench Verified 是 500 个经人工验证确实可解的任务子集，提供了更清晰的代码 Agent 能力度量。2025 年底，Claude Opus 4.5 在 SWE-bench Verified 上以 80.9% 领先，其次是 Gemini 3 Pro 的 76.8%。这个指标已经是各大公司发布新代码模型时的必报项目。
TheAgentCompany（新兴企业级 Agent 评测）
TheAgentCompany 是 NeurIPS 2025 发布的评测，专门衡量 LLM Agent 在真实职场任务上的表现，关注 Agent 是否能替代或加速实际工作。它的独特价值是直接和经济价值挂钩——不只问"任务完成了吗"，还问"这相当于多少人力成本"，是未来评测企业级 AI 落地效果的重要参考。
BFCL 成为 Tool Call 能力的工业标准
BFCL 专门测函数调用的正确性，对于把 LLM 和 API 集成使用的公司，这是确认模型不会产生错误调用的核心测试。随着 MCP（Model Context Protocol）在 2025 年成为 Agent 工具调用的事实标准，BFCL 的重要性进一步提升，现在几乎所有做 Agent 产品的团队都会把 BFCL 纳入回归测试。
