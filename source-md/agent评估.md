Agent 评测完整八股（2025 工业版）

---
一、为什么需要评估
避免被动循环 — 没有评估只能在生产环境发现问题，修一个 bug 引发新 bug，永远在救火。
加速开发 — 问题早期可见，修复成本最低，价值在整个生命周期中不断累积。
明确成功标准 — 迫使团队在开发前就把"什么叫好"写成可验证的具体定义，对齐所有人认知。
快速采用新模型 — 有评估体系的团队几天内就能量化新旧模型收益，快速决策是否升级。

---
二、评估的基本结构
任务（Task） — 最小测试单元，明确输入 + 成功标准。
试验（Trial） — 同一任务多次独立运行，消除随机性，做统计结论。
评分器（Grader） — 三类混用：代码评分器（客观可重现）、LLM-as-Judge（灵活处理主观）、人类评分器（黄金标准但昂贵）。
记录（Transcript） — 完整交互历史，含工具调用和推理过程，是排查问题的核心依据。

---
三、两大评估类型
能力评估 — 问"能做什么"，通过率起点低，探索能力上限，驱动提升。
回归评估 — 问"还能做以前做到的事吗"，通过率要求近 100%，防止功能退化。

---
四、评估指标体系（核心）
这是重点扩展区，工业界把指标分为四个维度：
4.1 结果维度
任务完成率（Task Completion Rate） — 最核心指标，Agent 成功完成任务的比例。必须用细粒度评分而非二元判断：0 = 完全失败、0.5 = 部分完成、1 = 完全成功。工业实践中还会进一步拆成里程碑完成率（Milestone KPI），把任务分解成子目标，统计"完成了几个子目标/总子目标数"，给出更细粒度的进度信号。
pass@k / pass^k — 处理非确定性的两把尺：pass@k 衡量能力上限（k 次中至少成功一次），pass^k 衡量稳定性（k 次全部成功）。工具类场景用 pass@k，流水线场景用 pass^k。
4.2 过程维度
轨迹评估（Trajectory Evaluation） — 不只看终点，看路径是否合理。核心方法是对比 Golden Trajectory（人工标注的最优路径），计算实际路径的偏差。关键子指标：步骤数（越少越好）、冗余工具调用数、是否走了弯路回溯。完成率高 + 轨迹差 = 能完成但效率低，成本高。
步骤效率（Step Efficiency） — 对比最优路径，评估 Agent 是否在用最少步骤完成任务。工业工具（如 DeepEval 的 StepEfficiencyMetric）会分析执行 trace，识别冗余操作（比如对同一查询调用了两次搜索工具）。
Action Advancement（行动推进度） — 最新研究提出的细粒度指标，对每一步单独打分：这一步是否真正让任务向目标推进了？而不是简单的"对/错"。
4.3 工具调用维度
这是 Agent 特有、普通 LLM 评测没有的维度，工业界非常重视。
工具选择准确率（Tool Selection Accuracy） — Agent 在给定任务下，是否选对了工具。对比实际调用的工具与期望工具列表（expected tools）计算准确率。
参数准确率（Parameter/Argument Accuracy） — 工具被正确选中后，传入的参数是否格式正确、语义合理。比如 API 调用的 JSON 参数是否符合 schema，SQL 查询是否语法正确。
工具执行成功率（Tool Efficacy） — 工具被正确调用后，是否真的对任务结果有帮助。即便调用形式正确，工具返回的结果可能无关或有误，这个指标捕捉"调对了但没用"的情况。
工具正确性综合分（Tool Correctness） — DeepEval 的 ToolCorrectnessMetric 将以上三者整合为一个分数，支持三种严格度：只看工具名 / 同时看参数 / 同时看输出。
4.4 规划维度
计划质量（Plan Quality） — Agent 面对任务时，制定的行动计划是否完整、合理、高效。用 LLM-as-Judge 评估计划的逻辑性和可行性（DeepEval 的 PlanQualityMetric）。
计划执行度（Plan Adherence） — Agent 实际执行时，是否忠实按照自己制定的计划行动。计划好但执行时偏离，是 Agent 的一类典型失败模式（DeepEval 的 PlanAdherenceMetric）。
4.5 输出质量维度
幻觉率（Hallucination Rate） — Agent 生成信息中，事实错误或凭空捏造内容的比例。是 LLM 类系统最关键的可靠性指标之一。检测方法：将输出与知识来源做对比，标注不支持的声明。
忠实度（Faithfulness） — 主要用于 RAG 场景，检测生成内容是否完全基于检索到的上下文，不引入上下文中没有的信息。
答案相关性（Answer Relevancy） — 回答是否真正回应了用户问题，而不是答非所问。
一致性（Consistency） — 对同样或相似的问题，是否给出一致的答案。同一问题换个说法就给出截然不同的答案，说明模型对该问题理解不稳定。
4.6 鲁棒性与安全维度
鲁棒性（Robustness） — 面对输入噪声、措辞变化、超出分布的场景时，Agent 表现是否稳定。评估方法：同一语义的问题用多种表达方式测试，看输出一致性（Metamorphic Testing）；用对抗性输入压测。
对抗攻击抵抗力（Adversarial Resilience） — 面对刻意设计的欺骗性输入（Prompt Injection、恶意指令、虚假信息），Agent 是否被误导。高风险场景（金融、医疗）尤其重要。
安全性（Safety） — 检测有害输出、毒性内容、偏见、隐私泄露等。工业工具如 Detoxify（毒性检测）、Giskard（安全扫描）是常见选择。
4.7 效率与成本维度
延迟（Latency） — 追踪响应时间，包括工具调用的耗时。延迟过高在交互类应用中会直接影响用户体验。
成本（Cost per Interaction） — 评估 Token 用量和 API 调用费用。在高并发场景中不加控制的 Token 消耗会使运营成本失控。
Token 效率（Token Efficiency） — 完成同等质量任务所消耗的 Token 数，越少越好，直接影响单次交互成本。

五、LLM-as-Judge 的偏见问题与缓解
Position Bias（位置偏见） — 偏向排在前面的答案。缓解：A/B swap，同一对答案正反各跑一次取平均。
Verbosity Bias（长度偏见） — 偏向更长的回答，即使内容稀疏。缓解：Prompt 中明确要求不因长度加分 + CoT 强制先分析再打分。
Self-Enhancement Bias（自我偏好） — 同一模型打分偏向自己的输出。缓解：引入多个不同模型交叉评估取平均。
缓解方法汇总： 多次评分取平均 → A/B swap → CoT 先分析后打分 → 多模型交叉评估。

六、不同智能体的评估方法
编码智能体 — 单元测试（正确性）+ 静态分析（代码质量）。基准：SWE-bench Verified（40% → 80%+）。
对话智能体 — 任务完成度 + 交互质量双维度，LLM 模拟用户做多轮测试。基准：τ-Bench。
研究智能体 — 信息来源权威性 + 声明有无出处 + 报告全面性和连贯性。
计算机使用智能体 — 真实环境 GUI 测试，验证最终状态，关注令牌效率与速度的平衡。

---
七、工业主流评估框架
DeepEval — 开源评估框架，像 Pytest 一样写 LLM 的单元测试。覆盖 Agent 指标（Task Completion、Tool Correctness、Plan Quality、Step Efficiency）、RAG 指标（Faithfulness、Contextual Recall）、安全指标，支持 CI/CD 集成，是目前工业界最常用的 Agent 评测工具之一。
RAGAS — 专注 RAG 系统的评估框架，核心指标：答案相关性、忠实度、上下文精确率、上下文召回率。轻量，易集成 LangChain/LlamaIndex。
LangSmith — LangChain 官方出品的追踪与评估平台，强项是可视化执行链路，适合 LangChain 生态。
Arize Phoenix — 开源 ML 可观测平台，支持 LLM trace 和评估，适合已有 MLOps 体系的团队。
Datadog LLM Observability — 企业级监控，可追踪整个多步骤 AI 流水线，内置幻觉检测、有害内容检测、PII 泄露检测，并与基础设施监控统一在一个平台。

八、从零构建评估的 8 个步骤
① 尽早开始 — 20-50 个真实失败案例就够启动，不要等完美。
② 从手动测试开始 — 人工感受"成功"是什么，固化后再自动化。
③ 编写明确任务 — 两个独立专家看同一案例能给出一致判断，才算标准清晰。
④ 平衡测试集 — 正向案例 + 边界/拒绝案例，避免乐观偏差。
⑤ 构建稳定环境 — 每次从干净初始状态启动，避免状态污染。
⑥ 精心设计评分器 — 评结果不评过程，允许部分得分（0/0.5/1）。
⑦ 检查记录 — 定期人工抽查，校准评分器防止漂移。
⑧ 监控饱和 — 通过率近 100% = 测试集失效，引入更难用例。

---
九、实践建议与常见陷阱
实践： 评估驱动开发（先定义测法再写功能）→ 领域专家参与定义成功标准 → 持续维护评测集 → 多层防护（自动化 + 生产监控 + 用户反馈）。
陷阱： 过早优化（等完美）→ 过度优化（只盯分数惩罚合理解法）→ 忽视维护（评测集漂移失效）→ 单一依赖（没有任何一种方法能覆盖所有问题）。

---
任务（Task）—— 出题人
就是一道具体的考题。它规定了：
- 输入是什么：「用户说：我的订单 #12345 还没到货，我要退款」
- 成功标准是什么：订单状态变为"已退款"，且用户收到确认消息
没有成功标准就不叫任务，叫聊天。

---
试验（Trial）—— 考同一道题考多次
把这道题让 Agent 跑 10 次。
为什么要跑多次？因为 Agent 每次回答都不完全一样（温度参数、模型随机性）。
- 第 1 次：成功退款 ✓
- 第 2 次：退款了但没发确认消息 ✗
- 第 3 次：成功退款 ✓
- ……
跑 10 次发现成功 7 次，得出结论：这道题通过率 70%。
只跑 1 次，碰巧成功了，你以为没问题——其实是运气。

---
评分器（Grader）—— 改卷老师
Agent 交卷之后，谁来判"这次算不算成功"？
- 代码评分器：直接查数据库，订单状态字段是否变成 refunded，是就给 1 分，不是给 0 分。客观、快速、准确。
- LLM-as-Judge：让另一个模型读 Agent 和用户的对话记录，判断"用户有没有被妥善安抚、解释是否清晰"，这种主观感受代码判断不了。
- 人类评分器：产品经理亲自读对话，给出最终裁定。最准但最慢最贵，通常只用来校准前两种是否评得准。
三种改卷老师各有盲区，所以通常组合用。

---
记录（Transcript）—— 考试录像
这 10 次考试，每次从头到尾发生了什么都被录下来：
- 第 3 步调用了退款 API
- 第 4 步查询了订单状态
- 第 6 步发送了确认消息
- ……
当第 2 次失败时，你翻出录像一看：原来 Agent 调用退款 API 成功了，但第 6 步发确认消息时工具调用参数格式写错了，导致消息没发出去。
没有录像，你只知道"失败了"，不知道"哪里失败的"。
