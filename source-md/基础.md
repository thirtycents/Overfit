## 💾 向量数据库的数据存储方式

传统的关系型数据库（如 MySQL）存储的是行和列中的结构化数据。而向量数据库，如 Milvus，则专注于存储和管理向量以及与之关联的元数据。

### 1. 核心存储结构

向量数据库通常存储以下两种关键数据：

|**存储项**|**内容**|**作用**|
|---|---|---|
|**A. 向量数据 (Vector Data)**|实际的高维数字数组（即 Embedding）。|它是相似度搜索的主体。数据库必须高效地读取和计算这些数组。|
|**B. 实体 ID & 元数据 (Metadata)**|每个向量都有一个唯一的 ID，并关联了原始数据（例如：这串向量对应的原始文本、图片 URL、创建时间、标签等）。|搜索完成后，系统通过这个 ID 找到并返回**原始、可读**的内容给用户。|

### 2. 物理存储方式

- **分片和分布式存储：** 向量数据库设计为**分布式系统**。数据会被拆分成小的**分片 (Shard)**，分布在多台机器上。这不仅保证了可扩展性（可以存储海量数据），也实现了并行查询（多台机器同时搜索）。
    
- **内存与磁盘：** 像 Milvus 这类系统会使用分层存储。
    
    - **索引结构**：为了实现快速查询，索引结构（HNSW 图、IVF 聚类信息）会被加载到**内存 (RAM)** 中，以达到纳秒级的访问速度。
        
    - **原始向量和元数据**：通常存储在**磁盘**上，只有在确定了相似的向量 ID 后才会被读取。

## 基础术语

### 1. 向量 (Vector) 和 Embedding

|**概念**|**解释**|**作用**|
|---|---|---|
|**向量 (Vector)**|一个由数字组成的**列表**或**数组**，代表数据在高维空间中的一个点。|它是所有向量数据库的基础存储单元。|
|**Embedding**|通过深度学习模型（如 BERT, Word2Vec, 或 Llama/GPT 系列模型）将**非结构化数据**（文本、图片、音频等）转换成**固定长度的数字向量**的过程和结果。|Embedding 捕获了原始数据的**语义信息**。语义上相似的数据，在向量空间中距离也会很近。|
|**维度 (Dimension)**|向量中数字的数量。例如，一个 768 维的向量有 768 个数字。|维度越高，通常能捕获的语义信息越丰富。|

### 2. 核心目标：相似度搜索 (Similarity Search)

|**概念**|**解释**|**作用**|
|---|---|---|
|**相似度搜索**|在数据库中找到与查询向量**最相似**（即距离最近）的 $K$ 个向量。|这是向量数据库的**核心功能**，支撑了语义搜索、推荐系统和 RAG 等应用。|
|**距离度量 ([[Metric]])**|计算向量之间**距离**或**相似度**的数学公式，例如：**L2 (欧氏距离)**、**COSINE (余弦相似度)**。|定义了“相似”的标准。|
|**ANN** (近似最近邻)|**目标**：以牺牲极小部分准确度为代价，实现**极快速**的相似度搜索。|实现了大规模、高维数据上的实时检索能力。|

### 3. 性能关键：索引 ([[Index]])

| **概念**             | **解释**                                   | **作用**                                            |
| ------------------ | ---------------------------------------- | ------------------------------------------------- |
| **索引 ([[Index]])** | 一种特殊的数据结构（如 HNSW 图、IVF 聚类），用于**组织**向量数据。 | **加速搜索。** 索引允许系统在查询时**跳过绝大部分无关的向量**，只检查少量可能相关的向量。 |
| **HNSW**           | 一种**基于图**的索引算法，提供极高的搜索速度和召回率。            | 目前最主流、性能最好的索引类型。                                  |
| **IVF_FLAT**       | 一种**基于聚类**的索引算法，通过将数据分组来缩小搜索范围。          | 适用于数据量巨大或对内存消耗敏感的场景。                              |
| **召回率 (Recall)**   | 衡量 ANN 搜索准确度的指标。召回率越高，找到的真实最近邻越多。        | 评估索引性能的关键指标。                                      |

### 4. 向量数据库的应用

|**应用**|**解释**|
|---|---|
|**RAG** (检索增强生成)|检索外部文档中最相关的向量片段，增强 LLM（大型语言模型）的回答准确性和时效性。|
|**语义搜索**|用户输入“我想看关于猫的视频”，数据库能理解背后的**意图**，并返回最相关的结果，而不是只匹配“猫”这个关键词。|
|**推荐系统**|查找与用户行为向量最相似的商品向量或内容向量。|

## ⚙️ 索引的真正作用：为什么需要它？

索引是向量数据库的“加速器”。它解决了一个核心问题：**如何在不比较所有向量的情况下，快速找到最近邻？**

### 案例：没有索引 vs. 使用索引

|**场景**|**搜索方式**|**耗时与效率**|
|---|---|---|
|**场景 A: 没有索引 (暴力穷举)**|您的查询向量必须和数据库中的 **所有 $1$ 亿个** 向量逐一计算距离。|**耗时极长。** 就像您在 1 亿本书中，一页一页翻找相似的内容。这是精确搜索 (kNN)，但不可用于实时应用。|
|**场景 B: 使用 HNSW 索引**|**HNSW** 预先建立了一个**图结构（路网）**。查询时，算法只需沿着预设的层级结构和最短路径导航，可能只需要计算 **几百或几千次** 距离。|**耗时极短。** 就像您使用 GPS 导航，只走主干道和最短路径。这是 ANN 搜索，效率极高。|

### 索引的两个关键作用

|**作用**|**解释**|
|---|---|
|**加速查找**|这是索引最主要的作用。它通过特定的数据结构（如 HNSW 的层级图）来**约束搜索空间**，避免无谓的计算。|
|**组织数据**|索引不仅是一个查找工具，也是数据的**组织方式**。它定义了向量之间的**拓扑关系**（谁和谁相邻，谁在哪个群组），这种关系是实现高效搜索的基础。|

**简而言之：**

**索引** 就是在数据加载时，通过复杂的计算，提前为所有向量创建的一张**高效导航地图**。查询时，数据库按照这张地图指引，直接奔向最有希望找到目标相似向量的区域，从而实现了 **ANN 目标**。
