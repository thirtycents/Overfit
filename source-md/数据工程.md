1.1 minhash+LSH
[图片]
minihash
minihash的核心思想是用一个很短的稠密向量（签名）来近似代表原来的大集合。
- 痛点：文章变成集合后，特征动辄几万个，直接算交并比太慢，且极占内存。
- 怎么做：
  1. 分词：把文档拆成 k -gram（如 shingles）构成集合；
  2. 哈希：用 n  个不同哈希函数，对每个集合计算哈希值，取最小值（Min）；
  3. 签名：n  个最小值构成 n -维签名向量。
- 魔法原理：数学上严格证明了，两篇文档的签名的重合率，近似等于两篇文章真实的 Jaccard 相似度。
 LSH 局部敏感哈希
单纯比较 MinHash 签名仍需 O(N2)  次对比。LSH 通过 Band 划分 进一步加速：
1. 分 Band：将 n  维签名分成 b  个 band，每 band 含 r  行（n=b×r ）；
2. 哈希：每个 band 单独哈希，决定文档进入哪个桶；
3. 候选对：仅当两个文档在至少一个 band 中碰撞，才视为候选相似对，进行精细比较。
整个流程
首先，通过 MinHash 将原始的庞大集合（如文档、图片特征）压缩成固定长度的短小签名，这样既能大幅降低数据规模，又能保证两个集合的 Jaccard 相似度可以用它们签名的相似度进行无偏估计；接着，利用 LSH 对这些签名建立索引，通过将签名分段并哈希到不同桶中的方式，确保相似的对象大概率落入同一个桶从而成为候选对，不相似的对象则被有效过滤；最终，只需对这个远小于全集的候选对集合进行精确的相似度计算和排序，即可在亚线性时间内快速找出与查询对象最相似的结果。
 minihash+ LSH的局限性 
1.1 纯“字面量”比对，完全是个“语义文盲”
1.2 对“短文本”极度不友好


1.2 Embedding 向量聚类去重
原理
- 文本向量化：使用预训练语言模型（如BERT、Sentence-BERT、SimCSE等）将每个文本转换为固定长度的向量（embedding）。这些向量在空间中的距离反映了文本的语义相似度——语义越接近，向量距离越小。
- 聚类：对所有文本的向量应用聚类算法（如KMeans、DBSCAN、HDBSCAN），将空间上接近的向量归为同一簇。聚类算法通过定义“邻近”规则（如欧氏距离、余弦距离）自动分组。
- 去重：在每个簇内选择一个代表文本（如中心点向量对应的文本，或随机选一个），作为该簇的唯一保留项，其余视为重复删除。聚类数量可由算法自动确定（如DBSCAN基于密度）或预先设定（如KMeans需指定K）。
怎么用
- 步骤1：生成embedding
选用适合的模型（如all-MiniLM-L6-v2兼顾速度与效果），为每条文本生成向量。注意文本长度、语言需与模型匹配，必要时先做清洗。
- 步骤2：选择聚类算法
  - KMeans：需预先指定聚类数K，适用于大致知道类别数且簇大小均匀的场景。可结合肘部法则等确定K。
  - DBSCAN/HDBSCAN：基于密度，无需指定簇数，能自动识别噪声点（孤立文本）。关键参数是邻域半径eps和最小样本数min_samples，对密度变化敏感但更灵活。
  - 层次聚类：可生成树状结构，按阈值切分，但计算复杂度高，不适合超大规模。
- 步骤3：设置相似度阈值
对于基于距离的聚类（如DBSCAN），阈值eps直接决定了语义相似的尺度：eps越小，簇内文本越相似；eps越大，越宽松。实际中常以余弦距离或欧氏距离作为度量，需要根据embedding空间分布调试。
- 步骤4：提取代表
从每个簇中提取代表性文本，可选取离簇中心最近的样本，或按某种策略（如最长的、质量最高的）保留一条。
- 步骤5：评估与调参
检查去重结果，调整聚类参数或embedding模型，平衡去重率与信息损失。
常见聚类算法的阈值含义
- KMeans：没有直接的相似度阈值，而是通过聚类数K间接控制。K越大，簇内相似度越高；K越小，簇内越宽松。常结合轮廓系数或业务经验定K。
- DBSCAN：参数eps（邻域半径）是关键阈值。在embedding空间（如归一化后的余弦距离），eps=0.3~0.6是常见试探范围。min_samples控制簇的最小规模，调高可减少噪声，但可能合并小簇。
- HDBSCAN：可设定min_cluster_size（最小簇大小）和cluster_selection_epsilon（距离截断），后者类似DBSCAN的eps，但HDBSCAN更鲁棒。

三种方案的本质区别
理解ColBERT，必须先把三种方案放在一起对比。
Bi-Encoder（你现在用的向量检索）
query  → Encoder → 一个向量q（1024维）
文档   → Encoder → 一个向量d（1024维）

相似度 = cosine(q, d)
核心特点： query和文档完全独立编码，两者之间没有任何交互。
文档向量可以提前算好存起来，查询时只需要算query的向量，然后做向量检索，速度极快。
缺点： 因为没有交互，精度有限。比如query是"茅台涨了多少"，文档里说"贵州茅台股价上涨8%"，Bi-Encoder可能识别不出来，因为"涨"和"上涨"、"茅台"和"贵州茅台"在向量空间里不一定足够近。

---
Cross-Encoder（你用的Reranker）
[query + 文档拼在一起] → Encoder → 直接输出相关性分数（0~1）
核心特点： query和文档深度交互，模型能看到两者的每个词，充分理解关系。
精度比Bi-Encoder高很多，但文档向量没法预计算。每来一个query，要把所有候选文档都和query拼在一起重新算一遍。
所以Cross-Encoder只能用在候选集已经很小（比如40个）的精排阶段，不能用来做全量检索。

---
ColBERT（介于两者之间）
这是关键，要理解清楚。
Bi-Encoder的问题是什么？
把整篇文档压缩成一个向量，信息损失太大。一篇文章有几百个词，压成1024维一个向量，很多细节丢失了。
ColBERT的思路：不压缩，保留每个token的向量。
query  → Encoder → 每个词一个向量  [q1, q2, q3]（比如query有3个词）
文档   → Encoder → 每个词一个向量  [d1, d2, d3, d4, d5]（文档有5个词）
然后用MaxSim计算相似度：
对query的每个词qi，找文档里和它最相似的那个词dj
把所有query词的最大相似度加起来，就是最终分数
举个例子：
query词：["茅台", "股价", "涨"]
文档词：["贵州茅台", "股票", "价格", "上涨", "8%"]

"茅台" → 和"贵州茅台"最相似 → 相似度0.92
"股价" → 和"股票"+"价格"里取最大 → 相似度0.85  
"涨"   → 和"上涨"最相似 → 相似度0.88

最终分数 = 0.92 + 0.85 + 0.88 = 2.65
这样"茅台"和"贵州茅台"、"涨"和"上涨"都能匹配上，精度比单向量高得多。

ColBERT为什么没用
存储计算
普通Bi-Encoder每个文档块存1个1024维向量。
ColBERT每个token存1个128维向量（ColBERT用128维，比Bi-Encoder低，但数量多得多）。
假设你的文档块平均200个token：
Bi-Encoder：1个文档块 = 1 × 1024维 = 1024个数字
ColBERT：   1个文档块 = 200 × 128维 = 25,600个数字
存储量是普通方案的25倍。
知识库有1万个文档块：
Bi-Encoder：1万 × 1024维 ≈ 40MB
ColBERT：   1万 × 200token × 128维 ≈ 1GB
PoC阶段这个存储成本不可接受，而且MaxSim计算也比单向量余弦复杂。
这是明确评估过的技术决策
不是不知道ColBERT，而是评估了之后主动选择不用。
当前知识库规模用Bi-Encoder+Cross-Encoder两阶段已经够用，如果未来知识库规模扩大到几千万文档，ColBERT的速度优势（相比Cross-Encoder）才会真正体现出来，那时候再迁移。

追问预案
Q：ColBERT的128维是怎么来的？
ColBERT模型训练时的输出维度设定的，比普通Bi-Encoder的1024维小，是为了在存储和精度之间平衡。维度小了存储少，但表达能力也弱一点。
Q：MaxSim为什么取最大值而不是平均值？
取平均值的话，文档里不相关的词也会拉低分数。比如query问"茅台股价"，文档里大部分在说宏观经济，只有一句提到茅台，平均相似度会很低。MaxSim只看最匹配的那个词，能捕捉到这种"局部相关"的情况，更符合检索的直觉。
Q：如果你的系统规模扩大，会考虑用ColBERT吗？
会重新评估。如果知识库扩大到千万级文档，Cross-Encoder精排的候选集生成速度可能成为瓶颈，这时候ColBERT在速度和精度之间的平衡点优势就体现出来了。但需要同时解决存储问题，比如用向量压缩或者只对高频查询的文档建ColBERT索引。
Q：Late Interaction是什么意思？
ColBERT的架构叫Late Interaction，意思是query和文档的交互发生在"晚期"——两边先各自独立编码（所以文档可以预计算），最后在MaxSim这一步才发生交互。相比之下Cross-Encoder是Early Interaction，一开始就把两者拼在一起，交互更充分但没法预计算。


Tf-idf  vs bm25
两者都是基于“关键词匹配”的经典文本检索算法，计算一个词与一篇文章的相关度。
[图片]
1. TF-IDF（基础版）
- 是什么： 词频（TF）× 逆文档频率（IDF）。
  - TF（词频）： 搜索词在文章中出现的次数越多，得分越高。
  - IDF（逆文档频率）： 这个词在所有文章中越稀有（比如“量子”比“的”稀有），权重越大。
- 致命缺点： 简单粗暴。词频没有上限（出现100次的分数远高于10次），且不考虑文章本身有多长，极容易偏向篇幅长、废话多的文章。
2. BM25（升级版/现代标准）
- 是什么： TF-IDF 的改良版。目前主流搜索引擎（如 Elasticsearch, Lucene）的默认标准。
- 核心改进（区别所在）：
  1. 词频饱和限制： 一个词出现几次后，得分就会“封顶”，不会随次数无限增加。有效防止了“关键词堆砌”。
  2. 文档长度归一化： 自带长短文本平衡机制。如果长短两篇文章都匹配了相同的词，较短的文章得分会更高（因为它的核心思想更集中）。
一句话总结：
TF-IDF 是早期的基础算法，容易被长文本和重复词“刷分”；BM25 是它的进化版，控制了词频上限并惩罚了注水长文，实际检索效果远超 TF-IDF。
